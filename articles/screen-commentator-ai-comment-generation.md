## 何を作ったか

画面をリアルタイムでキャプチャして、配信風のコメントをニコニコ動画みたいに流すmacOSアプリを作りました。

https://x.com/r1cA18/status/1895711413622780321

Swift / SwiftUI で作ったmacOSネイティブアプリで、ScreenCaptureKitで画面をキャプチャして、VLMでコメントを生成し、オーバーレイで表示する仕組みです。

見た目はシンプルで、ただコメントが画面上を流れるだけだと思ってしまいますが、実装を進める上で想像以上にAIアプリとして難しかったなと思います。

## 何が難しいのか

最初実装を検討した際は、画面キャプチャして、AIにコメント生成させて、オーバーレイで流せばいけそうだなーと思っていました。

すぐにClaude Codeにアイデアと軽い設計だけ渡して実装させたら、まあ動きはしました。

でもUXが全然良くなくて、ラグが目立つし、コメントの精度が低くて微妙でした。

冷静に考えると、このアプリが必要な要素は下記のことだとわかりました。

- 画面の内容を正確に理解する（テキストだけじゃなく、UIの構成、ユーザーが何をしているか）
- それに対して配信の視聴者っぽいコメントを生成する（「AIが作りました」感の少ないやつ）
- 複数のコメントを同時に、バリエーション豊かに生成する
- これを数秒間隔で常時リアルタイムでやり続ける

まともにやるとかなりちゃんと設計しないと求めているものが作れないなと感じました。

## そこで3つのパイプラインを試しました。

どこがボトルネックで、どういうパイプラインが最適なのかを特定するのが一番時間がかかりました。

### 1. OCR + 軽量LLM（画像をダイレクトで送らない方式）

最初に試したのは、スクリーンショットをそのままVLMに渡すのではなく、先にmacOSのVision frameworkでOCRをかけて画面のテキストだけ抽出し、それをもとにコメントを生成させるアプローチです。

[動画: OCR + Gemini 2.5 Flash Lite]

処理としては安定するし、速度も出る

画像を送らないからローカルLLMでも現実的に動くし、APIコストも安い

ただ、本質的な問題があって、 OCRでとってきた文字自体にノイズがだいぶ入ってしまっていました。

Claude Code が Cl3ude Cod8 になったり、メニューバーの文字が 01-IlulFEOFFII みたいな謎の文字列になったり、そのゴミがそのままコンテキストに入って、コメントが微妙なものになってしまいます。

わかっていたことではありますがOCRはテキスト抽出しかできないので、画面のレイアウト、UIの構成、ユーザーが今何をしようとしているか、みたいな画面の意味を理解することはしていません。

そのためコメントが画面の内容を踏まえてない感じになってしまってあんまり自然に見えませんでした。

### 2. VLM直接（ローカル軽量モデル）

次に試したのは、スクリーンショットをそのまま軽量なVLM（Gemma 3 4B）に投げるアプローチです。

[動画: Basic + ローカル Gemma 3 4B]

セキュリティ的にもコスト的にもこれが一番理想ではありました。

ただこれはモデルパワーが足りなかったです。

MacbookAirM4のユニファイドメモリ64GBですが、速度も出ないし、プロンプトも再現率悪いし、画像理解があまりできないしで微妙でした。

mood(funny) みたいなメタデータがそのままコメントとして表示されたり、以下コメント12個ですって説明文を出力してきたり、パースしようと思えばどうにかなりそうな気もしなくはないですが安定しないのと、画像理解もっとできるでかいモデルで試したかったのでやめました。

コメントの中身もプロンプトの指示に引っ張られすぎて、画面を見て反応しているというより、テンプレートを埋めてるだけみたいになってしまいます。

### 3. Gemini 3.0 Flash + 構造化ワンショット

最後に試したのが、現時点での正解でした。

4秒ごとにスクリーンショットを撮って、Gemini 3 Flashに投げる。1回のリクエストで、複数の方向性のコメントをJSON形式で構造化して出力させ、それを静的にパースして、次のスクショまでの間にコメントを順番に流していく方法です。

[動画: Smart + Gemini 3 Flash]

これが圧倒的に良かったです。

Swiftのコードを開いていたら「正規表現とか解読不能で草」「static letの羅列かっこいい」みたいなのがラグも少なくちゃんと出てきます。

Finderを開いたら「Downloadsの中身空っぽなの草」「サイドバーの項目多くね」みたいに、ちゃんと画面の中身を理解した上で、配信っぽいコメントを出せます。

やっぱりモデルパワーなんだなとはすごく思いました。

Gemini 3.0 flashは画像理解が比較的強いし、YouTubeとか配信文化みたいな知識もさっきまで検証していたモデルと比べて持っているので、「配信のコメントっぽく」って適当に言っても解釈が割とぽい感じにできていて、思い通りな感じにできました。

構造化出力も安定していて、JSONが崩れることがない。

## なぜ全コメントに画像理解が必要なのか

最初は、LLMにいくつかコメントを生成させつつ、シチュエーションを分類させて、あらかじめ用意したコメントプールからランダムで引っ張ってくるハイブリッド方式を考えていました。

静的処理とLLM処理のいいとこ取りで、速度もコストも抑えられるはず、という考えです。

しかし実際動かしてみると、画像理解に基づかないコメントが混ざった瞬間に体験がめちゃめちゃ崩れてしまいました。

「用意された文言そのまま流してるんだろうな」みたいなコメントが何個か混ざるだけでめちゃくちゃ萎えます。

逆に、全部のコメントが画面の内容をちゃんと踏まえていると、軽くでも没入できる感じがあります。

この差がかなり大きくて、結局全コメントがVLMの画像理解を通っているのがUXの最低ラインでした。

## オーケストレーションは必要か

最近のAIエージェントでは当たり前の、オーケストレーター + 専門ワーカーの構成も考えました。

高性能なオーケストレーターがいて、画像理解の専門モデルに並列で投げて、そこからコメントの方針を決めて、軽量モデルがコメント生成する、みたいな感じです。

結論、今のユースケースではあまり役立ってくれないなとなりました。

理由はシンプルで、処理を分割すると直列のレイテンシが増えてしまうからです。

4秒間隔でキャプチャしている中で、2段階のAPI呼び出しはそりゃそうですが現実的じゃないです。

そしてタスクの性質として、「画像 → 複数コメント」という変換は一つのモデルがワンショットでやるのが一番速い気がしています。

Gemini 3 Flashみたいな頭が良くて画像理解していて速いモデルに全部やらせるのが、今の最適解だと思いました。

責務分離してオーケストレーションすれば精度が上がるかというと、このケースではむしろレイテンシという別の問題を生むんだなーと。

## コメントのリアルさの問題

技術的に一番難しいのはコメントのリアルさだと思っています。

配信のコメントって、ジャンルや配信者によって空気感が全然違くて、かつ複数の性格、ロールでワンショットで生成させないといけません。

今の実装ではペルソナシステムを入れていて、Standard（一般視聴者）、Meme（ミームとか入れてくる人）、Critic（画面批評）、Instructor（指示厨）、Barrage（弾幕）みたいなペルソナを重み付きで混ぜていて、実験のためにUIで割合調整できるようにしています。

[画像: メインのウィンドウ]

いろんな方向性のコメントが混ざることでバリエーションが出ますが、まだAIっぽさが残る場面もだいぶあります。

## 画面を見てオーバーレイで表示できるAIの可能性

このアプリ自体は遊びで作ったもので、発展させる予定は特にないです、収益化はいろいろよくないと思うので

ただ、画面をリアルタイムで監視してオーバーレイ表示するという仕組み自体にはすごく可能性を感じました。

PCの操作を横でリアルタイムに説明してくれるアシスタントとか、サボってたら検知して教えてくれる監視ツールとか、レガシーなよくわかんないエンタープライズ向けのUIを触るときどこをクリックすべきか教えてくれるガイドとか、画面をVLMで常時理解して、リアルタイムでフィードバックするという基盤は使い方次第でいろいろ応用できそうだなと思っています（Microsoftがそんな感じのことWindowsでやろうとしてなかったっけ？）。

## まとめ

コメント流すだけに見える今回のアプリの実装は、リアルタイム画像理解、高速な構造化出力、自然なコメント生成、動的なコメント量調整、ペルソナシステムの組み合わせでした。

一番の学びは、見た目の要素が少ないアプリほど、UXの要求水準が高いということでした。

複雑なUIなら多少の粗は隠せるけど、今回のようなアプリでは、コメント一つ一つの品質がそのまま体験の質になってごまかしが効かない。

そしてその品質を担保するには、結局モデルパワーが必要でした。

軽量化やハイブリッド方式、オーケストレーションなどで工夫したが、全コメントが画像理解を通っていないとUXが成立しなかったです。
