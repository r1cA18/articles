## はじめに

自分は高専4年で、起きている時間のほとんどをClaude Codeに溶かしているような生活をしています。個人のナレッジベースをClaude Codeで管理し、タスク管理はNotion APIをClaude Codeのスキルから叩き、画像生成や動画生成のワークフローも日常的に回しています。典型的な「AIに使われている側」の人間だと思っています。

そんな自分が、**全国統一生成AI活用技能試験（全統AI）** の第1回を受けてきました。

この試験を見つけたのはXでした。最初は「見ず知らずの新しい試験、高いしめんどくさいな」と思っていました。ただ、生成AI、特にエージェントの活用レベルを人に一目で知ってもらえるような指標って今のところ存在しない気がしています。「こんだけ使ってるんだぞ」を証明できるものがないとは前々から思っていて、どうせなら受けてみようと思いました。

### 全統AIとは

全統AI（全国統一生成AI活用技能試験）は、2026年3月1日に第1回が実施された試験です。知識を問う選択式ではなく、実際にAIを使って成果物を作成・提出する実技試験という点が特徴的です。

- **制限時間**: 2時間
- **構成**: Section 1（6問、全員必須）+ Section 2（7テーマから1つ選択）
- **提出物**: レポート、画像、動画、Webサイト、設計書など多岐にわたる
- **受験料**: 13,000円（先行割などあり）

この記事では、**どう戦略を立てて臨んだか**、**何がうまくいって何がうまくいかなかったか**、そして**AIで試験を解くことで見えた本質的な問い**について書きます。

## AIの試験をAIで解く

試験を受けてまず率直に感じたのは、「こんなに思考力が必要な問題を人間が2時間で理解しきるのは無理じゃない！？」ということでした。

内容は言えないので公式サイトのサンプル問題を見てほしいのですが、学術系のドメイン知識が必要だったり、マーケティングの職業レベルでの理解が必要だったり、とにかく専門的です。1セクションを深く理解して解くだけで3時間はかかりそうだと判断しました。6セクション + 選択問題を2時間で！普通に考えて人間がボトルネックになります。

そのため、**自分は問題を深く理解して解く人ではなく、ツールやエージェントに方針を出すオーケストレーター/レビュワーとして動く**ことを選ばざるを得ませんでした。

### 役割分担

| 担当            | 作業内容                                                                                  |
| --------------- | ----------------------------------------------------------------------------------------- |
| **Claude Code** | 問題解析、回答Markdown作成、draw.io図生成、HTML/CSS実装、PDFビルド                        |
| **Codex CLI**   | 各セクションのレビュー・精査（`/post-review`コマンド）                                    |
| **自分**        | 画像生成（nanobanana2）、動画生成（Sora 2.0）、スライド生成（Gemini）、最終チェック、提出 |

Claude Codeが「手を動かす人」、Codex CLIが「レビュワー」、自分は「AIが苦手なクリエイティブ生成 + 最終判断」という分担です。

### ワークフロー

試験開始から提出までの流れはこうでした:

1. **問題のスクレイピング**: Claude Codeに試験ページを読み取らせ、全問題を構造化
2. **戦略立案**: 事前に解いていたサンプル問題の経験をもとに、解く順番と方針を決定
3. **並列作業**: Claude Codeが回答を生成している間に、自分は画像生成・動画生成を並行して進める
4. **レビュー**: Codex CLIの`/post-review`（自作のレビュースキル）で各セクションの回答を精査
5. **修正・提出**: レビュー結果をもとに修正し、提出

Claude Codeが6セクション分の回答を作っている間に、自分は別のツールで画像や動画を生成していました。待ち時間をほぼゼロにできたのは大きかったです。

## 使ったツールスタック

### Claude Code（メイン）

試験全体のオーケストレーションを担当。問題の解析から回答の作成、draw.ioでの図生成、HTML/CSSの実装、MarkdownからのPDFビルドまで、テキストベースの成果物はほぼすべてClaude Codeに任せました。

普段からスキルやワークフローを整備していたので、「問題を読んで → 計画を立てて → 回答を作成して → レビューに回す」という流れを自然に指示できました。日常の延長線上で試験を受けている感覚です。普通の試験はこんなことできないのでなんか悪いことをしてる気分。。。

### Codex CLI（レビュー）

Claude Codeが生成した回答の精査に使いました。`codex exec --full-auto --sandbox read-only`で、問題文と回答を照合させ、PASS/WARN/FAILの3段階で評価させます。

Claude Codeに自分の回答をレビューさせると甘くなりがちだったので、別のモデル（Codex）に批判的にレビューさせることで客観性を担保する狙いです。

### nanobanana2（画像生成）

画像の生成に使用。ここは自分の手で直接プロンプトを書いて生成しました。
スライド生成にも使いました。

### Sora 2.0（動画生成）

画像から短いプロモーション動画を生成するのに使用。Image-to-Videoの問題で活用しました。

### 使い分けの判断基準

シンプルに言うと、**テキストベースの成果物はClaude Code**、**ビジュアル系はそれぞれ専門のツール**、**品質チェックはCodex**です。Claude Codeは文章・コード・構造化データの生成は強いですが、「見た目の良さ」の判断がまだ弱かったり、トークン使用が気になるのであんまり芸術系には使えません。そのため、ビジュアル系は自分の目で確認しながら専門ツールで作りました。

## 具体的な問題と解き方

### 抽象的な構造を見出す問題

あるセクションでは、異なる領域の共通構造を見出す課題が出ました。Claude Codeに対話形式で構造を引き出させ、draw.ioで比較図を生成しました。ただし手書きの図が求められる問題もあり、AIだけでは完結しない設計でした。ここはiPadで自分で描きました。

draw.ioでの図生成は「それっぽいもの」はできるけど、**レイアウトの美しさがコントロールしにくい**のが課題でした。フォントサイズ、ノード間のスペーシング、色の統一感など、「見やすい図」にするためのプロンプト指定がまだ確立できてないなーと思いました。

### 画像生成・動画生成

画像生成と動画生成を用いるセクションもありました。

ここは**自分が最も手を動かしたセクション**です。nanobanana2で画像を生成し、Sora 2.0で動画を作りました。

感じたのは、**アート系のプロンプトエンジニアリングはコード系と全く別物**だということです。
コード系はコンテキスト（仕様書、既存コード、テスト）を正しく渡せば高精度な出力が得られます。
しかしアート系は一回のプロンプトにこだわらないといけなかったり、たくさん試行しないといけなかったり、一回一回が重いし大変でした。

## うまくいかなかったこと

### draw.ioの図生成

MermaidやCSVからdraw.ioの図を生成できるのですが、出力される図のレイアウトが綺麗にならないことが多かったです。ノードが重なったり、矢印が交差したり。「見やすい図」を生成するためのフォーマット指定の工夫がもっと必要だと感じました。

### アート系プロンプトの難しさ

コード系のプロンプトでは「こういう仕様で、こういうテストが通るように」と明確に指示できます。でもアート系は「もうちょっとファンタジーっぽく」「もっと空気感を出して」みたいな抽象的な指示になりがちで、試行回数が増えます。

### AIに目を持たせること

Claude Codeはテキストの品質判断は得意ですが、画像や図の「見た目の良さ」を判断するのはまだ弱いです。「この図は見やすいか？」「このレイアウトは直感的か？」という視覚的な品質評価は、結局人間がやるしかなかったし、綺麗にできるskillsをもっと使いこなしたい。

## AIで試験を解いて感じたこと

### 「トークンをバカバカ使って頑張っただけ」感

正直、試験中ずっとこの感覚がありました。

AIに回答を生成させ、別のAIにレビューさせ、自分は画像生成ツールを回しながら軽く理解しながら最終チェックをする。
作業としては効率的（？）なんですが、**自分が何かを深く理解した実感がない**。他人事感が増して、全然楽しくない。

ただ、今まで培ってきた各AIツールの知見、Claude Codeのワークフロー設計、スキルシステムの構築経験が全部活きている実感はありました。
**AIを道具として使いこなす能力**は多少発揮できていたとは感じますが、それは理解とは全然別のものな気がしてしまってます。

### ドメイン理解をするしかない！！

AIは検索ツール・生成ツールとしては優秀ですが、生成物の良し悪しを評価するには、そのドメインの深い理解が必要だとすごく感じました。

あまり詳しくない分野のセクションでは、生成された回答が「合っているのか間違っているのかすらわからない」状態で提出するしかなかった。

**幅広い知識を持っていること自体が、AIのレビュー品質を決定する。** これが一番の学びだった気がします。

### AIは幅を広げるが深さは変えない

ビジネス的に言えば、制限時間内に成果物を量産できるのは価値があるのかな？と思います。
1つの問題に3時間かけて完璧な回答を作るより、2時間で全問それなりの品質で提出するほうが実践としては正解かもしれない。

しかしドメイン理解を深めようと思ったら、AIがあっても変わらず時間をかけて一つを極める必要があると思っています。
結局、研究や学習の本質は変わっていなくて、AIは**できることの幅**を広げてくれるけど、**理解の深さ**は変えてくれないんじゃないかなと思いました。

理想は、**自分で知る意欲を持ちながら生活し、それを記録して、自分の能力もエージェントも同時に成長させていく**ことだと感じてます。
AI時代の学びってそういうことなんだと思います。

## おわりに

結果として、制限時間の10分前に全問提出を完了できました。

この試験を通して一番考えさせられたのは、「**AIの試験をAIで解くこのアプローチは、試験が測ろうとしているものの本質に合っているのか？**」ということです。

正直よくわかりません。
ただ少なくとも、AIを使いこなすスキル、複数のツールをオーケストレーションする能力、生成物の品質を短時間で判断する目は試されていたと思います。そしてそれは、普段の生活で培ってきたものがそのまま出るもので、付け焼き刃ではどうにもならない試験でした。

第1回の試験なので評価基準がどうなるかは未知数ですが、「AI活用の実力を測る試験」という試み自体は面白いと思っています。
結果が出たらまたなにか書きます。
